### 新增

- **全拟真假人系统（从零原创，营造在线人气 + 丰富世界生态，默认关闭、零现网影响）**：批量生成「没有网络连接的假人玩家」在世界里自动活动——上下线、巡逻走动、打怪、保活、死亡复活、捡金币、喊话、PK，让在线列表 / 地图 / NPC 脚本 `USERCOUNT` 都显示出人气。核心是利用引擎既有的「无连接玩家」能力（`new 玩家实例(角色, null)` 自动置 `是否假人=true`、全引擎推包 `网络连接?.发送封包` 对 `null` 连接空安全），主要新增「调度 + 存盘隔离」两层。
  - **存盘隔离（头号设计要点，杜绝污染 `Data.db`）**：假人 `角色数据` 用带参构造生成合法角色后**立即 `删除数据()` 整株摘出全部持久表**（角色 + 初始背包 / 装备 / 技能），常驻内存对象池复用、任何时刻都不停留在 `角色数据表`——因为「关服重扫 `检测物品数据(添加到数据表:true)`」「每秒 `保存数据()` / 周期 `导出数据()`」「GM 整理 / 清理重排索引」三条落库路径都只扫表内对象，假人不在表即永不落盘。假人共用一个账号（`new 账号数据` 后同样 `删除数据()` 出账号表、仅保留对象引用）；运行期假人只捡金币（`物品过滤` 为空 → `玩家自动战斗` 只收编号 < 10 的货币、走货币不入物品表）、初始装备创建时已随角色摘表，外加下线时 `产出清理` 把背包 / 装备逐个 `删除数据()` 作第二道防线。

- **单实例锁：阻止在同一目录重复启动游戏服务器（防双开写坏存档）**：原先在同一目录重复双击会起两个游戏服务器进程，二者并发写同一份客户数据 `Data.db`、抢绑同一监听端口——轻则端口绑定失败崩溃，重则存档损坏。`Program.Main` 最前置新增单实例守卫：以「运行目录路径的 SHA256」为键创建全局命名 `Mutex`(`Global\YH_GameServer_Instance_<目录哈希>`)，同目录的第二个实例取不到锁即弹窗提示并退出；不同目录键不同、互不阻塞，**仍允许把整个服务端复制到不同目录多开**。`Mutex` 存静态字段随进程存活，进程退出(含崩溃)由 OS 自动释放命名对象、不会卡死下次启动；取锁机制本身一旦异常(如 Global 命名空间 ACL)一律放行启动——宁可不拦也不因锁机制本身打不开服。`账号服务器` 为独立进程、用专属名称前缀区分，不受影响([Program.cs](游戏服务器/Program.cs))

- **启动数据加载并行化**：海量模板小文件（技能 5581 / 物品 4445 / NPC 4215 / 地图 2695 等约 1.9 万个 `.txt`）原先在 `序列化类.反序列化` 里逐文件串行 `File.ReadAllText` + 多轮整串 `.Replace` + `JsonConvert`，是启动慢的主因之一。两个 `反序列化` 重载的内层循环改为 `Parallel.ForEach`：非泛型重载本就用线程安全的 `ConcurrentQueue`；泛型重载把非线程安全的 `List` 换成 `ConcurrentQueue`，并补齐逐文件 `try/catch`+记日志（单个损坏文件不再中断整批加载）。多核下接近线性加速；`数据表` 写入仍在 `反序列化` 返回之后单线程进行，无竞态。代价：返回数组顺序不再确定（现有调用方均按对象自身字段建索引，不依赖顺序）
  - **怪物爆率二次 IO 并行化**：`游戏怪物.载入数据` 反序列化完 2323 个怪物文件（`Npc数据/怪物数据/`）后，原本还有一个串行 `for` 逐怪物 `加载怪物爆率`（`File.OpenText` 逐行解析，含 `#CALL`/`@` 递归再开文件），Lever A 未覆盖。每个怪物只写自己的 `怪物掉落物品`、彼此独立，改为 `Parallel.ForEach` 并行加载（每怪物一次、爆率文件存在才打开，`怪物爆率/` 共 582 个文件 + `#CALL`/`@` 递归）；`数据表.Add` 保留在并行之后单线程执行
  - **快速 `Point` 转换器（点集解析提速的关键）**：耗时探针显示瓶颈不在文件数而在文件体积——`地图区域`（164 文件 / 56MB）与 `怪物刷新`（515 文件 / 85MB）合计 141MB，主体是 `范围坐标` 里数百万个 `"x, y"` 坐标点。Newtonsoft 默认走 `System.Drawing.PointConverter`（文化相关解析，慢）。新增 `快速坐标转换器 : JsonConverter<Point>`（`Split`+`int.TryParse(ReadOnlySpan)` 直解，写回仍为 `"x, y"`、与既有文件格式兼容，并兼容 `{"X":..,"Y":..}` 对象形式），注册进 `全局设置`，直接提速这数百万个点的反序列化
  - **跳过无谓的整串替换**：`序列化类.反序列化` 两个重载在替换前先判定 `$type` / `Assembly-CSharp` 是否存在，不含旧标记的纯坐标点集文件跳过全部 18 轮 `.Replace`；含 `$type` 的技能节点文件照常替换（修正了首版仅判 `Assembly-CSharp` 漏掉技能短类型名导致的 `Error resolving type`）。注：.NET 的 `string.Replace` 在无匹配时返回原串、不分配内存，故此项本身收益有限，主要是省去对超大文件的重复扫描
  - **巨型文件负载均衡**：`地图区域` / `怪物刷新` 目录里混着 3.4MB 巨型文件与大量小文件，默认 `Parallel.ForEach` 分区可能把大文件压在一个核上拖成长尾。`序列化类.反序列化` 两个重载改为先按体积降序排序、再用 `Partitioner.Create(files, loadBalance: true)` 动态分区，大文件优先被领取、小文件填满空闲核
  - **加启动耗时探针**：`系统数据网关` 每个 loader 的日志追加 `耗时 Xms` 并在末尾打印总耗时；`SMain` 启动加载三阶段（系统数据 / 脚本系统 / 客户数据）各打印耗时与总耗时，便于定位剩余瓶颈
  - `系统数据网关` 的 58 个 loader 外层**仍保持串行**：存在 4 条加载期跨类型依赖（`传送法阵`→`游戏地图` / `装备属性`→`随机属性` / `游戏套装`→`游戏物品` / `地图道具`→`游戏怪物`），整批 `Parallel.ForEach`（已注释的 `LoadData`）会撞 NRE / 集合已修改，故当年退回串行是正确的，本次未动

- **传永武技签到成功即「游戏卡死、只能强退」（签到成功漏发协议刷新包）**：玩家在「传永武技签到」NPC 点签到、且满足条件（本期特权 4/5/7 且当日未签）成功签到后，客户端直接卡死必须强退。根因在协议复用——武技签到这套封包**复用了官方月卡签到的操作码**：`传永武技签到`=366=`g2c_sync_monthly_card_checkin_info`（月卡信息）、`传永武技回执`=365=`g2c_monthly_card_checkin`（月卡签到确认）。官方月卡协议的下发顺序恒为「先 366 信息、再 365 确认」，客户端的 365 处理器依赖先收到 366 把签到 UI 的状态机建好。而 `玩家实例.传永武技签到()` 的成功分支**只发了 365 回执、漏发 366 刷新**，客户端拿着 365 去操作未初始化的月卡签到状态 → 卡死。对照参考源（同款客户端、可正常运行）证实：其成功分支是「先发 `传永武技签到{签到次数=传永武技}`(366)、跳出后再发 `传永武技回执{回执编号=0}`(365)」两个都发。修复：在成功分支于 365 之前补发更新后的 366（`签到次数=this.传永武技`、签到后 `能否签到=false`），顺序与官方月卡协议及参考源一致。为何**日常签到不受同类影响**：日常签到走的是专用操作码 288/289（`g2c_sync_checkin_info` / `g2c_checkin`），不存在「先信息后确认」的顺序依赖，故即便它在 YH 里同样只发应答(`每日签到应答`)、未刷新同步包(`同步签到信息`)也能正常工作，本次按「最小改动、不动正常功能」原则未动它（[地图类\玩家实例.cs](游戏服务器/地图类/玩家实例.cs)）

- **LF 换行的成长配置文件被整份当成「第 1 行」致服务器启动崩溃**：`角色成长` 静态构造函数用 `Regex.Split(..., "\r\n")` 按**字面量 CRLF** 切行解析 `成长属性.txt` / `玩家升级经验.txt` / `玩家升级战力.txt`。当数据文件是 LF（`\n`）换行（编辑器另存、跨平台传输等很常见）时文中无 `\r\n` 序列，`Split` 切不开，**整个文件被当作一行**，于是 `值 = "10\n2\n3\n20…"`、`long.Parse` 抛 `FormatException`，在 `static cctor` 中被包成 `TypeInitializationException` 冒泡为未处理异常，服务器起不来；错误框「第 1 行」后跟着整份文件内容正是此症状的铁证。修复：两处切行的分隔符由 `"\r\n"` 改为 `"\r\n|\r|\n"`，对 CRLF / LF / 单 CR 三种换行均正确切分且行尾不残留 `\r`（`成长属性.txt` 按 `\t` 分列也因此更稳）。本仓库自带的数据文件本是 CRLF、不受影响，此改为公开引擎用户的 LF 数据文件兜底（`角色成长.cs`）
- **关服点「X」卡死 / 未响应、关服要等 20 秒以上**：游戏服务器 `服务循环` 的收尾段（`主程.循环.cs`）写死了两段 `Thread.Sleep(10000)`（清理物品后一次、强制保存后一次），与全量 `强制保存`+`导出数据`（序列化落盘 + GZip 备份）叠加，关一次服最少干等 20 秒；而当前在用的 `SMain.OnClosing` 在 UI 线程里自旋 `while (主程.主线程 != null)` 等收尾跑完，期间窗口无法重绘、被 Windows 标记「未响应」，表现为"卡住"。把两段 `Thread.Sleep(10000)` 各改为 `Thread.Sleep(500)`（其前后的清理/保存/导出均为同步调用、返回即已完成，长睡眠纯属冗余保险），关服时长由 20s+ 降到约 1s + 真实落盘耗时；同时把收尾末行日志改为输出 `服务器已成功关闭, 关服保存耗时:X.X 秒`，方便直接从系统日志判断落盘真实开销
- **关服收尾与后台定时自动保存的写盘竞态**：定时自动保存 `保存数据库()`（`主程.服务.cs`）走 `Task.Run` 在后台执行 `导出数据()`，期间 `自动保存中=true`；若关服恰在该后台任务进行中触发，主线程收尾段的 `清理物品 / 强制保存 / 导出数据` 会与之并发，可致 ① 后台遍历数据表时撞 `清理物品` 的删改（集合已修改异常）② 两路并发写同一对象的 `MemoryStream` 损坏序列化 ③ 两个 `导出数据` 并发 `File.Create/Move` 同一批数据文件而崩溃。修复：收尾段开头新增 `while (主程.自动保存中)` 等待（含 30s 超时兜底 + 超时日志），等当前这次后台自动保存落盘完成再继续；此时主循环已退出、不会再触发新的自动保存，故只需等这一个，正常无后台保存在跑时该循环零开销（`主程.循环.cs`）
- **`游戏商店` 加载缓慢（84 个商店文件 / 669KB 却耗时约 350ms）**：`游戏商店.载入数据` 把整段固定字节的 CRC32 放进 `for (i < array.Length)` 循环算了「字节数」次、累加进会溢出且最终被覆盖的 `num`，构成 O(n²)（`游戏商店.cs`）。改为单次 `计算类.CRC(array)`；`商店文件效验` 仅作客户端缓存版本令牌（`玩家实例.cs:8028` 回传比对、客户端不自算 CRC），改后部署首次开商店重新同步一次即稳定、无害。该 loader 由约 350ms 降到个位数 ms
- **登录卡死("点登录没反应")**：账号服务器登录成功响应字段数与客户端不一致，导致登录后界面不跳转、无任何提示
  - 根因：MED-E/PROTO-01「登录响应不再回显密码哈希」只改了客户端（`登录界面.cs` case "0" 改为严格校验 `array.Length==4`），账号服务器 `网络通信.cs:279` 的注释虽已加上「不再回显」，但代码仍把 `array[3]`（密码哈希）拼进响应，实际发出 5 段 `包号 0 账号 哈希 区服列表`。客户端收到 5 段 `!= 4` → 静默 break → 界面停滞
  - 修复：账号服务器登录响应删除多余的密码哈希字段，固定为 4 段 `包号 0 账号 区服列表`，与客户端对齐（同时兑现 PROTO-01 不回显哈希的安全意图）。注册/改密/门票三类响应经核对字段数本就一致，无需改动
- **登录器请求超时无提示**：`ServerCfg.txt` IP/端口错误或账号服务器未响应时，30s 超时计时器只静默解锁界面，表现为"点了没反应"。改为弹窗提示「连接服务器超时」并指引检查服务器与配置（`登录界面.cs` 新增 `登录请求超时`，替换 `用户界面计时.Tick` 原绑定的 `用户界面解锁`）
- **中文 / 非 ASCII 安装路径下 Lua 脚本加载失败**：服务端放在含中文的目录（如 `公开引擎`、`游戏服务器`）时，`require "init"` 等报 `module 'init' not found`，且报错里的中文路径显示为 `������` 乱码方块
  - 根因：NLua 把含中文的 `package.path` 按 UTF-8 交给 Lua，而 Lua 5.4 原生 `require` 底层用 C 运行库 `fopen` 按系统 ANSI 代码页（简体中文为 GBK/936）打开文件，无法识别 UTF-8 字节序列的中文路径
  - 修复：`游戏脚本.cs` 新增静态方法 `加载lua模块` + 一个走 .NET `File.ReadAllText` 的 Lua 模块搜索器（插入 `package.searchers` 第 2 位，优先于默认 path 搜索器）。读文件完全在托管侧完成、只把源码交给 `load`，彻底绕开原生 `fopen`；入口 `main.lua` 同步改用 `File.ReadAllText` + `DoString` 加载。中文与纯英文路径均可正常加载
- **中文变量名 Lua 脚本无法编译（魔改 Lua 未接入构建）**：脚本系统大量使用中文标识符（`主程`、`计算类` 等），标准 Lua 5.4 词法分析器不接受非 ASCII 标识符，修好上面的路径问题后 `init.lua` 暴露出 `unexpected symbol near '<\228>'`
  - 根因：作者准备的、支持中文标识符的魔改 `lua54.dll` 一直留在 `游戏服务器/` 根目录，但从未接入构建；KeraLua（NLua 依赖）始终加载 NuGet 自带的标准 Lua（`runtimes/win-x64/native/lua54.dll`）
  - 修复：`游戏服务器.csproj` 新增 `AfterTargets="Build"` 与 `"Publish"` 的 MSBuild Target，构建/发布后自动用根目录的魔改 `lua54.dll` 覆盖输出根目录与 `runtimes/win-x64/native/` 下的标准版。`clone` + `build` 即自动启用，无需手动替换 dll
- **C07 安全守卫回归：短定长封包被一律踢线（玩家几乎寸步难行）**：第五轮 C07 为堵「变长封包总长字段填 0/1/2/3 致 `取封包` 游标零推进、接收回调 `while(true)` 无限重解析」的预鉴权 DoS，加了 `if (value2 < 4) throw` 断连守卫。但该检查被放在 `value2 = (变长 ? 读头部总长 : 注册表长度)` 这条合并语句**之后**，于是把**所有注册长度 < 4 的定长包**也一并误杀——而长度 2/3 的客户端定长包遍地都是：`查询地图路线`(530)、`打开角色背包`(598)、`玩家退出登录`(183)、`玩家每日签到`(191)、交易确认/锁定/解锁/结束(151~154)、摆摊系列(103/105/106) 等数十个核心操作。任一这类包经 [客户网络.cs](游戏服务器/网络类/客户网络.cs) 的 `catch → 尝试断开连接(e)` 直接踢线，表现为玩家点地图/开背包/签到即掉线（系统日志刷 `封包总长非法(变长封包头部不足4字节)! 封包编号:530 总长:2`）。修复：把 `< 4` 检查收进**变长分支内部**（仅当注册 `长度 == 0` 才读头部并校验），定长包 `value2` 取自可信注册表、游标必 `Skip(value2≥1)` 推进、不存在死循环，故不再被波及；变长包的 DoS 防护原样保留（`游戏封包.cs` `取封包`）

## 版本约定

宽松遵循 [语义化版本](https://semver.org/lang/zh-CN/)：

- **MAJOR** — 网络协议或数据格式不兼容（客户端需同步升级）
- **MINOR** — 新功能 / 大规模重构 / 安全加固
- **PATCH** — Bug 修复 / 小幅调整

当前处于 0.x 阶段，所有 0.X.0 版本可能包含破坏性改动。
